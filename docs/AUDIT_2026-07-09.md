# Torrorèndum — Full Production Audit (Performance + API Correctness)

**Date:** 2026-07-09 · **Scope:** entire Go/chi/Postgres backend (`cmd/`, `internal/`, `migrations/`, `public/`) · **Build:** clean (`go build`, `go vet`, `go test` all pass)

This audit had two interlocked goals: (1) find performance bottlenecks that threaten a bursty, share-driven, single-VPS deployment, and (2) exhaustively exercise every endpoint and every use case to surface correctness/validation misses. Both were pursued together — the black-box test pass and the code audit cross-checked each other, and each found bugs the other missed.

## How this was done (reproducible)

- **Server + DB stood up locally.** Throwaway Postgres 16 in podman; the real server binary booted against it (`DB_*`/`PORT`/`ADMIN_TOKEN` env, migrations auto-run on boot).
- **Realistic volume seeded** to make aggregation endpoints work under pressure: **5,051 users, 200,050 `Results` rows, 20,097 `UserEloSnapshots`** (a plausible "went viral" state), then `ANALYZE`.
- **Empirical measurement**, not just reading: per-endpoint latency profiling, concurrency race reproduction, rate-limit bypass tests, `EXPLAIN (ANALYZE, BUFFERS)` on hot queries, and status-code probing of every error path.
- **Two independent analysis passes** — a black-box endpoint × use-case matrix (43 routes, 227 cases → `docs/API_TEST_MATRIX.md`) and an 8-dimension code audit with adversarial verification (32 findings, each refuted-or-confirmed by two independent reviewers).
- **A permanent regression suite** was built from the findings: `test/e2e/` (self-contained; boots its own DB + server, asserts every route + every confirmed bug as a guard). See `test/e2e/README.md`.

Everything below that says "measured" or "reproduced" was executed against the running server, not inferred.

## Part A — Performance

### A1. `/premsa` recomputes full-table aggregations, uncached — and collapses under load **[HIGH]**
`internal/repository/postgres_press_stats.go` — every hit runs `MostVotedTorro` (1 full `Results` scan), `BiggestRiser` (a `UNION ALL` = **2** full scans), and `ClosestDuel` (another scan). `EXPLAIN ANALYZE` at 200k rows: ~90ms *per scan*, three parallel-worker seq-scans, no index used, no caching.

- **Solo latency measured: 480–600ms.** Under **50 concurrent: p95 13.8s, max 14s** (static `/` stayed at 126ms). Each request grabs 2–3 parallel workers; 50 concurrent × workers saturates every core and the 25-connection pool. This is a public, journalist-facing page — one press release linking it is a self-inflicted DoS, and `WriteTimeout=15s` means the next increment of load returns truncated 500s.
- **Fix:** compute these staleness-tolerant stats on a timer into an in-memory value (or a materialized view refreshed every N minutes); add `CREATE INDEX idx_results_winner ON "Results"("Winner")` for `VotesForTorro`.

### A2. `/wrapped` self-joins `Results` on the unindexed `Pairing` column **[HIGH]**
`postgres_wrapped_stats.go:55` — full-table scan per user request. **Fix:** `CREATE INDEX idx_results_pairing ON "Results"("Pairing")`. This is the single highest-leverage index add — it also helps A1's `BiggestRiser`/`ClosestDuel` (both join `Results` by `Pairing`) and any future per-pairing query.

### A3. No compression anywhere **[HIGH]**
`server.go` middleware chain has no `middleware.Compress`. **Measured:** `main.css` = 172,112 bytes uncompressed; a `gzip`-accepting request gets the same 172,112 bytes. **Fix:** `r.Use(middleware.Compress(5, "text/html","text/css","application/javascript","application/json","image/svg+xml"))` — ~172KB→~30KB CSS, ~23KB→~6KB HTML, one line.

### A4. No caching on embedded `/public/*` assets **[HIGH]**
`server.go:149` — the `embed.FS`-backed `http.FileServer` emits no `Cache-Control`, no `ETag`, and `embed.FS` reports a zero `ModTime` so `Last-Modified`/`If-Modified-Since` can't help either. ~19MB of images re-downloaded on every visit. **Fix:** wrap the file server to set `Cache-Control: public, max-age=31536000, immutable` (assets are baked into the binary) or set an `ETag` from the build id.

### A5. `/stats` fires ~8 redundant round-trips **[MEDIUM]**
`stats_handler.go:135` re-queries per-class vote counts already present in the `Users` row it just loaded (`ClassVotes` jsonb). **Fix:** parse `ClassVotes` once in Go; drop the per-class queries.

### A6. `/reveal` ships every qualifying user's `ClassVotes` to Go per request **[MEDIUM]**
`postgres_persona_stats.go:124` computes a percentile by reading + JSON-parsing the whole cohort each hit. **Fix:** cache the cohort distribution, or compute the percentile in SQL.

### A7. PNG render pipeline: unbounded concurrency + slow compression **[HIGH / MEDIUM]**
`sharecard/canvas.go` — each card allocates a full RGBA (~8.3MB for the large cards) with **no global concurrency limit** (`canvas.go:69`), and `png.Encode` uses `DefaultCompression` which is ~68% of render CPU (`canvas.go:89`). Four public PNG routes (`/share/card`, `/wrapped/card`, `/reveal/card`, `/press-kit/card`). **Fix:** gate all four behind a `GOMAXPROCS`-sized semaphore (return 503 when full); switch to a shared `png.Encoder{CompressionLevel: png.BestSpeed}`; pool the RGBA backing arrays.

### A8. Latency baseline (200k results, single request, measured)
`/` 3ms · `/classes` 6ms · `/classes/1/vote` 20ms · `/leaderboard` 20ms · `/stats` 22ms · `/history` 16ms · `/wrapped` 90ms · `/reveal` 35ms · **`/premsa` 480–600ms** · `/api/*` 7–15ms. Everything except `/premsa` (and secondarily `/wrapped`) is healthy; those two are the whole performance story on the read side.

## Part B — Correctness & validation

### B0. `/sitemap.xml`, `/robots.txt`, `/llms.txt` are unreachable dead code **[HIGH]** *(black-box-only find)*
`server.go:159–161` registers these three routes *with* their extension, but `middleware.URLFormat` (registered globally) rewrites chi's routing path to strip a trailing `.ext` before the router matches. **Reproduced — every URL form 404s:** `/sitemap.xml`→404 *and* `/sitemap`→404 (request `/sitemap.xml` → routing path stripped to `/sitemap` → no route `/sitemap` exists → 404; request `/sitemap` → stays `/sitemap` → still no match). So the `robotsTxt`/`sitemapXML`/`llmsTxt` handlers can't be hit at any URL — dead code. By contrast `/share/card` is registered *without* the extension, so both `/share/card` and `/share/card.png` resolve (verified). The code's own comment warns about this exact trap for `.png` but the fix was never applied to the SEO routes. For an SEO-focused campaign, crawlers get no sitemap and no robots.txt. **Fix:** register the three routes without the extension (mirroring `/share/card`), or exempt these paths from `URLFormat`.

### B1. Bad ids → `500` instead of `404` on public routes **[HIGH]**
`handler.go:162` (`GET /classes/{id}/vote`) and `handler.go:250` (`POST /pairings/{id}/vote`) wrap a `NotFound` repo error in `ErrInternal`. **Reproduced:** `/classes/999/vote`→500, `/classes/abc/vote`→500, `POST /pairings/<nonexistent>/vote`→500. All `Id` columns are `VARCHAR(36)` (not native `uuid`), so a malformed id is indistinguishable from a nonexistent one — both should be 404. **Fix:** branch on `NotFoundError` (as `torroDetail` already does) → 404; keep 500 only for genuine driver failures. A shared repo-error→HTTP mapper would fix this class everywhere.

### B2. Negative `?offset` on `/history` → `500` **[MEDIUM]**
`history_handler.go:61` — a negative offset flows straight into SQL `OFFSET`, Postgres rejects it. **Reproduced:** `/history?offset=-5`→500 (while `offset=abc` and overflow correctly default to 0). **Fix:** `if offset < 0 { offset = 0 }` (or 400).

### B3. Unbounded `?size` on bracket create **[MEDIUM]**
`bracket_handler.go:584` — `size` is validated for power-of-two but not bounded, so a huge value drives a multi-GB allocation before the "not enough torrons" check. (Also: `size=1` passes `isPowerOfTwo` and slips to a misleading downstream error.) **Fix:** clamp to `min(64, active torrons in class)`; reject `<2`.

### B4. Inconsistent unknown-class contract **[MEDIUM]**
`GET /api/leaderboard/class/999` → **404**, but `GET /leaderboard?category=999`, `GET /api/user/leaderboard/class/999`, and `GET /embed/leaderboard?classId=999` → **200 + empty**. Same conceptual error, three different behaviors. **Fix:** validate class id once, consistently.

### B5. Gate-threshold inconsistency **[LOW]**
`/wrapped`, `/reveal`, `/api/user/leaderboard/global` gate at **50** votes; the personal-global `/leaderboard` gates at **25** (`getMinVotesForClass` default). A 25–49-vote user is unlocked on one surface, locked on the others. **Fix:** one shared threshold constant.

### B6. Raw driver error text leaked to clients **[LOW]**
`repository/error.go:87` `errUnknown` wraps the raw `pq`/driver error; `domain/error.go` puts everything after the first colon into the JSON `message`. **Fix:** log raw server-side, return a generic message + numeric code.

## Part C — Security

### C1. Vote manipulation is wide open (two independent bypasses) **[CRITICAL]**
The whole point of the per-user vote limiter is to stop scripted ELO manipulation. Both layers fail:
- **Per-user limiter (`server.go:76`):** keyed on the `torrons_user_id` cookie, but a request with **no cookie** makes `UserMiddleware` mint a brand-new user — so every cookieless (or cookie-rotated) request is "user 1 of its own bucket" and is never throttled. **Reproduced:** 50 concurrent cookieless votes all returned 200.
- **Global per-IP limiter (`server.go:49/54`):** `middleware.RealIP` trusts `X-Forwarded-For`/`X-Real-IP` from *any* client with no trusted-proxy allowlist, so `httprate.KeyByIP` keys off an attacker-controlled header. **Reproduced:** fixed IP → 100 pass then 429; **rotating XFF → 140/140 pass.**

Combined, an attacker can cast unlimited votes for any torró and arbitrarily set the public ranking — which, for a contest with brand/legal stakes, is the highest-impact bug here. **Fix:** enforce vote dedup in the DB (`UNIQUE(UserId, Pairing)` or an IP-bucket + pairing constraint); drop/limit `RealIP` to a real trusted proxy that overwrites inbound forwarding headers; consider a signed cookie or lightweight challenge before the first vote.

### C2. Unauthenticated unbounded DB growth **[MEDIUM]**
`middleware.go:68` — every cookieless request creates a persistent `Users` row (compounds C1's bypass: an attacker spraying cookieless requests also inflates the table). **Fix:** defer user creation until first state-changing action; add a reaper for zero-vote anonymous users older than N days.

### C3. Identity is an unauthenticated bearer of a UUID **[LOW, by design — documented]**
`GET /api/user/stats` returns any user's stats given their `torrons_user_id`. UUIDs aren't trivially enumerable, but this is worth knowing before any of them leak. Recorded as a guard in the suite so future hardening (signed cookies) is noticed.

Injection surface: **none found.** All `Id` columns are `VARCHAR(36)` with fully parameterized queries; `dietaryFilterSQL` builds only from fixed boolean flags. A `'; DROP TABLE ...` id is inert (behaves as a nonexistent id); verified the table survives.

## Part D — Concurrency

### D1. ELO lost update on `Torrons.Rating` **[CRITICAL]** *(the anchor finding)*
`handler.go:result()` reads both torró ratings via `GetTx` (plain `SELECT`, **no `FOR UPDATE`**), computes new ELO in Go, and writes `UPDATE "Torrons" SET "Rating"=$2` (absolute). Concurrent votes on the same torró read the same pre-image and overwrite each other. **Measured** (2 torrons reset to 1500, 50 concurrent votes for one winner): result **1555.78** vs. serialized-correct **1771.07** — ~79% of the movement lost; all 50 `Results` rows written (so the corruption is invisible in vote counts). **Fix:** `SELECT ... FOR UPDATE` on both rows in a deterministic id order, or make the write relative (`SET "Rating" = "Rating" + $delta`) with the delta computed from a locked read.

### D2. Same lost-update pattern on `UserEloSnapshots` **[LOW]**
`handler.go:366` — per-user personalized rating/vote-count is also read-modify-write with an absolute UPDATE and no row lock. Lower impact (needs the *same* user voting concurrently) but the same remedy class.

### D3. Concurrent round-closing bracket votes **[MEDIUM]**
`bracket_handler.go:379` — `check-then-advance` runs at READ COMMITTED with no lock. Two concurrent votes that together close a round either (A) both miss completion → round stalls until the next vote/admin force-advance, or (B) both build round R+1 → the loser hits the `idx_bracket_matches_slot` unique index, its whole tx (including a valid vote) rolls back → **500 + silently dropped vote**. **Fix:** `SELECT pg_advisory_xact_lock(hashtext(bracketId))` right after the vote insert to serialize the advance.

## Part E — Lifecycle, build, data model

- **E1 [HIGH] Graceful shutdown is not graceful.** `server.go:296` hands `httpServer.Shutdown` the already-cancelled `srv.ctx` → returns immediately without draining; `api.go:117` then closes the DB pool while requests may still be in flight (`Server.Shutdown` is fire-and-forget). **Fix:** pass a fresh `context.WithTimeout(context.Background(), 20s)` to `Shutdown`, and make `api.Shutdown` block on drain before `db.Close()`.
- **E2 [MEDIUM] `--skip-migrations` doesn't skip.** `main.go` guards its own migration pass, but `api.NewDatabaseConnection` (`api.go:341`) runs the embedded `m.Up()` unconditionally. Migrations run on every boot regardless of the flag (double-migration path). **Fix:** thread the flag through, or consolidate to one mechanism.
- **E3 [MEDIUM] Version stamping is broken.** `Makefile:8` ldflags target `github.com/krtffl/torrons/internal/version` but the module is `github.com/krtffl/torro` and the package is `github.com/krtffl/torro/version`. `make build`/`dist` never stamp version/branch/revision/built. **Fix:** correct the `-X` paths; add a CI version smoke test.
- **E4 [MEDIUM] First-boot pairing seeding is non-transactional with swallowed errors.** `api.go:259` inserts thousands of pairings one-by-one, `continue`-ing on error → a hiccup leaves a permanently partial set that `count<1` won't re-trigger. Also no `UNIQUE(Torro1,Torro2,Class)`, so a two-instance boot double-seeds. **Fix:** seed per class in one transaction; add the unique index + `ON CONFLICT DO NOTHING`; advisory-lock the seed.
- **E5 [MEDIUM] Healthcheck doesn't touch the DB.** `healthcheck.go:19` returns 200 while Postgres is down. **Fix:** `db.PingContext` with a short timeout → 503 on failure.
- **E6 [LOW] Docker startup logs a misleading "Migration failed" ERROR** every boot from the redundant file-based path (`main.go:45`). **Fix:** pick one migration mechanism.
- **E7 [LOW] Down migrations violate FKs** (`000012...down.sql` and others delete parents before children) — a rollback on a populated DB fails. **Fix:** delete children first / add `ON DELETE CASCADE`.

**Deliberately deferred (documented, not fixed) — product decisions:**
- **C2** every cookieless request creates a `Users` row — closing this fully (defer creation to first vote, or reap zero-vote anonymous users) changes user/analytics semantics; it's a product decision. The now-un-spoofable global 100/min-per-real-IP limit bounds the abuse vector in the meantime.
- **Vote manipulation, full closure** — the per-user vote limiter's cookieless-rotation vector is now backstopped by the un-spoofable per-IP limit; hard closure (DB vote-dedup / captcha / signed cookie) is a product decision (dedup would change the re-vote UX).

At this point every originally-identified finding is either fixed or is a deliberate, documented product-policy call. There is no remaining known bug in this audit's scope.
